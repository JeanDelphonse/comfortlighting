1. Overview
This document defines the requirements for a Lead Workflow Progress Diagram displayed on the Lead Detail page of the ComfortLighting.net Lead Inventory System. The diagram provides an at-a-glance visual representation of where a lead currently sits in the 12-stage sales pipeline — from New Lead through to Closed Won, Closed Lost, or On Hold.
The diagram is rendered as an interactive SVG pipeline stepper. The current stage is highlighted. Completed stages are visually checked off. Terminal stages (Closed Won, Closed Lost) and the special On Hold stage have distinct visual treatments. Clicking any stage node on the diagram advances or retreats the lead to that stage, updating the PROGRESS field in the database without navigating away from the Lead Detail page.

3. Workflow Transition Rules
The pipeline is primarily sequential (stages 01–09) with two terminal exits (Closed Won, Closed Lost) and one reversible pause (On Hold). The following rules govern valid stage transitions:

3.1 Forward Transitions (Normal Flow)
Any stage 01–09 can advance to the immediately next stage by clicking the next stage node on the diagram
Skip-forward is permitted — a rep may jump multiple stages (e.g., New Lead → Proposal Sent) if the prospect situation warrants, with a confirmation prompt
From Contract Sent (09), the lead may advance to Closed Won (10) or Closed Lost (11)

3.2 Backward Transitions (Retreats)
Any active stage (01–09) can retreat to any earlier active stage — for example, Proposal Sent may return to Contacted if the deal restarts
Closed Won and Closed Lost are terminal — they cannot advance or retreat without an explicit Admin override
On Hold can be moved to any active stage 01–09 when the prospect re-engages

3.3 On Hold — Special Behavior
Any active stage (01–09) can be moved to On Hold at any time
On Hold does not appear in the main pipeline stepper — it is displayed as a separate indicator below the diagram
When returned from On Hold, the lead resumes at the stage it occupied before being placed on hold (the previous_progress value is preserved)

3.4 Terminal Stage Rules
Closed Won and Closed Lost are permanently highlighted in their terminal colors — the stepper does not show any further progression arrow
Admin users may override a terminal stage to re-open a lead (e.g., returning Closed Lost to Contacted for a new attempt) — a mandatory reason note is required

PROGRESS FIELD ALIGNMENT
The 12 pipeline stages map directly to the PROGRESS controlled vocabulary in the leads table. Any change made via the diagram widget writes the new value to leads.progress immediately via an async PATCH request — the same field used for filtering, reporting, and the Sales Activity expense tracker.
The diagram is always in sync with the PROGRESS field — they are the same data, displayed two ways.

4. Diagram Design Specification
4.1 Layout — SVG Pipeline Stepper
The diagram is a horizontal SVG stepper rendered across the full width of the Lead Detail page. On screens narrower than 900px (tablets, small laptops) the diagram wraps into a two-row layout. On screens below 600px it collapses into a vertical stack.

4.4 Arrow Persistence — Historical Arrows
All past transition arrows are permanently rendered and never removed. As a lead advances through the pipeline, each from_stage → to_stage transition leaves a persistent colored arrow on the diagram. This gives an at-a-glance history of the deal's movement without opening the stage history log.
Completed arrows are rendered in muted green (#5A9E7A) — distinct from the active blue arrow so the current transition stands out clearly
Historical arrows do not animate — they are static to distinguish them from the active transition
If a lead retreated backward (e.g., Proposal Sent → Contacted), the retreat is shown as an amber curved arrow below the track spanning from the retreat origin to the retreat destination, with an arrowhead pointing left
All retreat arrows are also persistent — if the lead later re-advances past the same stages, both the original forward arrow, the retreat arrow, and the re-advancement arrow are all visible simultaneously
Arrow history data is sourced from the lead_stage_history table — the macro loops over all history rows to render each arrow

4.6 Stage Change Confirmation Modal
Clicking a stage node opens a small confirmation modal (not a full page): 'Move this lead to {Stage Name}?'
The modal shows the current stage, the target stage, and a brief description of what the stage means
Two buttons: Confirm and Cancel
For skip-forward jumps (more than one stage), an additional warning is shown: 'You are skipping X stages — are you sure?'
For moves to Closed Won or Closed Lost, the modal includes an optional text field: 'Reason / closing notes'
For Admin override of a terminal stage, the reason field is required

5. Placement on the Lead Detail Page
The diagram is placed in a dedicated card component near the top of the Lead Detail page, immediately below the lead title bar (Company Name + ACTION badge) and above the main field grid. This ensures the pipeline position is the first piece of context the rep sees when opening a lead.

6. Functional Requirements
6.1 Diagram Rendering
The diagram is rendered as an SVG element generated by a Jinja2 macro — built using inline SVG <line>, <polygon>, and <path> elements for precise arrow control
The macro receives the current PROGRESS value AND the full stage history list (all rows from lead_stage_history for this lead)
Each stage-to-stage transition in the history list produces a persistent directional arrow on the diagram
SVG dimensions: 100% width, viewBox height 160px (increased from prior spec to accommodate retreat arrows below the track)
The diagram is accessible: each stage node includes aria-label, role='button', and tabindex='0' for keyboard navigation
Color is not the sole differentiator — arrows have distinct stroke widths, dash patterns, and arrowhead sizes per state

6.2 Clickable Stage Nodes — Stage Change Interaction
Each stage node is clickable (desktop) and tappable (mobile — native touch events)
On hover, the outbound arrow from the hovered node briefly highlights in blue (#2E75B6) to preview what the arrow would look like if that transition were taken
Click fires a JavaScript event that opens the confirmation modal
On modal Confirm: JavaScript sends PATCH /leads/<id>/progress with { progress: 'Stage Name' }
On success: the Flask route re-renders the full diagram macro (including all history arrows) and returns the SVG HTML — JavaScript replaces the diagram DOM node without a page reload
The newly drawn active arrow animates in with a brief draw-on effect (stroke-dashoffset animation, 0.4s) to highlight the new transition
On failure: the modal closes with an error toast — the diagram and arrows are unchanged

6.3 On Hold Badge
If leads.progress = 'On Hold', the main stepper shows the last active stage (leads.previous_progress) with a yellow pulsing ring
An ON HOLD badge (amber, ⏸ icon, 'On Hold since {date}') appears directly below the stepper
A Remove Hold button on the badge moves the lead back to its previous_progress stage
The On Hold option is available in all stage confirmation modals as an alternative action: 'Put On Hold Instead'

6.4 Stage History Tooltip
Hovering over any completed stage node shows a tooltip: '{Stage Name} — {date reached} — {days spent}'
Stage history is stored in a new lead_stage_history table — one row per stage transition
This gives reps and managers visibility into how long a lead has spent at each stage

6.5 Days in Stage Counter
Below the current stage node, a small counter shows: 'In {Stage Name} for {N} days'
This is calculated from the most recent entry in lead_stage_history for the current stage
If N > 14 days for any active stage, the counter text turns amber (warning) — the rep may need to act
If N > 30 days for any active stage, the counter turns red — the lead is flagged as stale

6.6 Arrow Persistence — History Arrows
All arrows are persisted and visible at all times — they are never cleared or reset when the stage advances
On page load, the Flask route passes the full lead_stage_history queryset to the Jinja2 macro alongside the current lead
The macro loops over every history row and renders the corresponding arrow for each from_stage → to_stage pair
Forward arrows (from_stage index < to_stage index) are rendered on the main track between the two nodes, colored in muted green (#5A9E7A) for completed transitions
The most recent forward arrow (the active transition) is rendered in bold blue (#2E75B6) with an animated arrowhead pulse
Backward retreat arrows (from_stage index > to_stage index) are rendered as curved amber arrows below the node track, spanning from the origin node to the destination node with an arrowhead pointing left (←)
When both a forward arrow and a retreat arrow exist between the same two nodes, both are displayed simultaneously — the forward arrow on the track, the retreat arrow below it — giving a full picture of the deal's oscillation
Each persisted arrow includes a data-from and data-to attribute so JavaScript can reference and animate them individually

7.3 Migration SQL
migration_workflow_diagram.sql
-- Migration: Lead Workflow Diagram support
-- Database: comfortlighting  |  User: comfort

-- New columns on leads table
ALTER TABLE leads
  ADD COLUMN previous_progress VARCHAR(50) NULL DEFAULT NULL AFTER progress,
  ADD COLUMN stage_entered_at  DATETIME   NULL DEFAULT NULL AFTER previous_progress;

-- Backfill stage_entered_at for existing leads (use updated_at as proxy)
UPDATE leads SET stage_entered_at = updated_at WHERE stage_entered_at IS NULL;

-- New stage history table
CREATE TABLE lead_stage_history (
    id           INT AUTO_INCREMENT PRIMARY KEY,
    lead_id      INT         NOT NULL,
    from_stage   VARCHAR(50) NULL,
    to_stage     VARCHAR(50) NOT NULL,
    changed_by   INT         NOT NULL,
    changed_at   DATETIME    DEFAULT CURRENT_TIMESTAMP,
    reason       TEXT        NULL,
    days_in_stage DECIMAL(7,1) NULL,
    CONSTRAINT fk_lsh_lead FOREIGN KEY (lead_id)
        REFERENCES leads(id) ON DELETE CASCADE,
    CONSTRAINT fk_lsh_user FOREIGN KEY (changed_by)
        REFERENCES users(id) ON DELETE RESTRICT,
    INDEX idx_lsh_lead    (lead_id),
    INDEX idx_lsh_changed (changed_at)
);

8.1 PATCH /leads/<id>/progress
routes/leads.py  — progress update route
@leads_bp.route('/leads/<int:id>/progress', methods=['PATCH'])
@login_required
def update_progress(id):
    lead      = Lead.query.get_or_404(id)
    data      = request.get_json()
    new_stage = data.get('progress', '').strip()
    reason    = data.get('reason', None)

    VALID_STAGES = [
        'New Lead','Call Scheduled','Contacted','Quote Requested',
        'Proposal Sent','Follow Up','Negotiation','Contract',
        'Contract Sent','Closed Won','Closed Lost','On Hold'
    ]
    if new_stage not in VALID_STAGES:
        return jsonify({'error': 'Invalid stage'}), 400

    # Terminal stage override requires Admin role
    terminal = {'Closed Won','Closed Lost'}
    if lead.progress in terminal and not current_user.is_admin:
        return jsonify({'error': 'Admin required to re-open a terminal lead'}), 403

    from datetime import datetime, timezone
    now = datetime.now(timezone.utc)

    # Calculate days spent in current stage
    days_in = None
    if lead.stage_entered_at:
        delta = now - lead.stage_entered_at.replace(tzinfo=timezone.utc)
        days_in = round(delta.total_seconds() / 86400, 1)

    # Write history row
    history = LeadStageHistory(
        lead_id       = lead.id,
        from_stage    = lead.progress,
        to_stage      = new_stage,
        changed_by    = current_user.id,
        changed_at    = now,
        reason        = reason,
        days_in_stage = days_in
    )
    db.session.add(history)

    # On Hold handling
    if new_stage == 'On Hold':
        lead.previous_progress = lead.progress
    elif lead.progress == 'On Hold':
        lead.previous_progress = None

    lead.progress       = new_stage
    lead.stage_entered_at = now
    db.session.commit()

    # Return updated SVG fragment for DOM replacement
    svg_html = render_template('leads/_progress_diagram.html', lead=lead)
    return jsonify({'success': True, 'diagram_html': svg_html}), 200

static/js/pipeline_diagram.js  — arrow-aware version
// ── Pipeline Diagram — stage click + arrow interaction ─
const STAGES = [
  { name:'New Lead',        desc:'Lead created — no contact attempted yet.' },
  { name:'Call Scheduled',  desc:'A call or meeting has been scheduled.' },
  { name:'Contacted',       desc:'Initial contact made with the prospect.' },
  { name:'Quote Requested', desc:'Prospect has requested pricing or a quote.' },
  { name:'Proposal Sent',   desc:'Formal proposal delivered to the prospect.' },
  { name:'Follow Up',       desc:'Rep following up after proposal delivery.' },
  { name:'Negotiation',     desc:'Prospect actively negotiating terms or price.' },
  { name:'Contract',        desc:'Contract being prepared or reviewed internally.' },
  { name:'Contract Sent',   desc:'Contract delivered to prospect for signature.' },
  { name:'Closed Won',      desc:'Deal closed — contract signed, project confirmed.' },
  { name:'Closed Lost',     desc:'Deal lost — competitor, declined, or went dark.' },
  { name:'On Hold',         desc:'Lead paused — prospect requested delay.' },
];

const CSRF = document.querySelector('meta[name=csrf-token]')?.content;
let pendingStage = null;

document.addEventListener('DOMContentLoaded', () => {
  attachNodeHandlers();
  attachArrowHoverPreviews();
});

// ── Attach node click + keyboard handlers ─────────────
function attachNodeHandlers() {
  document.querySelectorAll('.node-circle[data-stage]').forEach(node => {
    node.addEventListener('click',   () => openModal(node.dataset.stage));
    node.addEventListener('keydown', e => { if (e.key==='Enter') openModal(node.dataset.stage); });
  });
  const removeHold = document.querySelector('.remove-hold-btn');
  if (removeHold) removeHold.addEventListener('click', unhold);
}

// ── Arrow hover preview ───────────────────────────────
// When the user hovers over a future node, briefly tint
// the future arrows leading to it in blue to preview direction.
function attachArrowHoverPreviews() {
  document.querySelectorAll('.node-circle--future').forEach(node => {
    node.addEventListener('mouseenter', () => {
      const targetStage = node.dataset.stage;
      document.querySelectorAll('.arrow-future').forEach(arrow => {
        if (arrow.dataset.to === targetStage || !arrow.dataset.to) {
          arrow.style.stroke = '#2E75B6';
          arrow.style.strokeDasharray = 'none';
          arrow.style.opacity = '0.7';
        }
      });
    });
    node.addEventListener('mouseleave', () => {
      document.querySelectorAll('.arrow-future').forEach(arrow => {
        arrow.style.stroke = '';
        arrow.style.strokeDasharray = '';
        arrow.style.opacity = '';
      });
    });
  });
}

// ── Open confirmation modal ───────────────────────────
function openModal(targetStage) {
  const diagram      = document.getElementById('pipeline-diagram');
  const currentCircle= diagram.querySelector('.node-circle--current') ||
                       diagram.querySelector('.node-circle--terminal-won, .node-circle--terminal-lost');
  const currentStage = currentCircle?.dataset.stage;
  const fromIdx      = STAGES.findIndex(s => s.name === currentStage);
  const toIdx        = STAGES.findIndex(s => s.name === targetStage);
  if (fromIdx === toIdx) return;

  pendingStage = targetStage;
  const isTerminal = ['Closed Won','Closed Lost'].includes(targetStage);
  const isSkip     = Math.abs(toIdx - fromIdx) > 1;
  const isRetreat  = toIdx < fromIdx;

  document.getElementById('modal-company').textContent =
    diagram.dataset.company || 'this lead';
  document.getElementById('modal-from-badge').textContent = currentStage || '—';
  document.getElementById('modal-to-badge').textContent   = targetStage;
  document.getElementById('modal-stage-desc').textContent =
    STAGES[toIdx]?.desc || '';
  document.getElementById('modal-skip-warning').style.display   = isSkip    ? '' : 'none';
  document.getElementById('modal-retreat-warning').style.display = isRetreat ? '' : 'none';
  document.getElementById('modal-reason-group').style.display   = isTerminal ? '' : 'none';

  // Preview: highlight the specific arrow for this transition
  highlightPreviewArrow(currentStage, targetStage);

  new bootstrap.Modal(document.getElementById('stageModal')).show();
}

// Preview arrow highlight — shows what the new arrow will look like
function highlightPreviewArrow(fromStage, toStage) {
  // Temporarily add a preview dashed blue arrow over future arrows
  const svg  = document.querySelector('.pipeline-svg');
  if (!svg) return;
  const old  = svg.querySelector('.arrow-preview');
  if (old) old.remove();
  // The actual preview is handled by CSS hover state — no extra DOM needed
}

// ── Confirm stage change ──────────────────────────────
document.getElementById('modal-confirm-btn')?.addEventListener('click', async () => {
  if (!pendingStage) return;
  const leadId = document.getElementById('pipeline-diagram').dataset.leadId;
  const reason = document.getElementById('modal-reason')?.value || null;

  const res  = await fetch(`/leads/${leadId}/progress`, {
    method:  'PATCH',
    headers: { 'Content-Type':'application/json', 'X-CSRFToken': CSRF },
    body:    JSON.stringify({ progress: pendingStage, reason })
  });
  const data = await res.json();

  bootstrap.Modal.getInstance(document.getElementById('stageModal')).hide();

  if (data.success) {
    // Replace diagram with server-rendered version (includes all history arrows)
    document.getElementById('pipeline-diagram').outerHTML = data.diagram_html;
    attachNodeHandlers();
    attachArrowHoverPreviews();
    // The new active arrow has class arrow-draw-on → CSS animates it in
    showToast(`Moved to: ${pendingStage}`, 'success');
  } else {
    showToast(data.error || 'Could not update stage.', 'danger');
  }
  pendingStage = null;
});

// ── Remove On Hold ────────────────────────────────────
async function unhold() {
  const leadId = document.getElementById('pipeline-diagram').dataset.leadId;
  const res    = await fetch(`/leads/${leadId}/unhold`, {
    method:'POST', headers:{'X-CSRFToken':CSRF}
  });
  const data = await res.json();
  if (data.success) {
    document.getElementById('pipeline-diagram').outerHTML = data.diagram_html;
    attachNodeHandlers();
    attachArrowHoverPreviews();
    showToast('On Hold removed — lead resumed.', 'success');
  }
}

@leads_bp.route('/leads/<int:id>', methods=['GET'])
@login_required
def view_lead(id):
    lead = Lead.query.get_or_404(id)

    # Load full stage history for arrow rendering
    stage_history = (
        LeadStageHistory.query
        .filter_by(lead_id=id)
        .order_by(LeadStageHistory.changed_at.asc())
        .with_entities(
            LeadStageHistory.from_stage,
            LeadStageHistory.to_stage,
            LeadStageHistory.changed_at
        )
        .all()
    )

    return render_template('leads/view.html',
                           lead=lead,
                           stage_history=stage_history)

12. Acceptance Criteria

AC #
Criterion
AC-01
The pipeline diagram renders on the Lead Detail page immediately below the title bar, above the field grid.
AC-02
All 12 stages are displayed in the correct order. On Hold is shown as an amber badge below the diagram, not as a node in the main pipeline row.
AC-03
Each stage-to-stage transition is represented by a directional arrow with a visible arrowhead pointing from the from_stage node toward the to_stage node (→). There are no plain connector lines — only arrows.
AC-04
Completed (historical) arrows are rendered in muted green (#5A9E7A) with a solid 3px shaft and filled arrowhead. They are visually distinct from the active arrow.
AC-05
The most recent (active) from_stage → to_stage arrow is rendered in bold blue (#2E75B6) with a 4px shaft, larger arrowhead, and a draw-on animation (stroke-dashoffset) that plays once when the stage is updated.
AC-06
Future arrows (stages not yet reached) are rendered as dashed gray lines (#CED4DA) with faint open arrowheads.
AC-07
All completed history arrows persist on the diagram permanently — they are never cleared or overwritten when the lead advances to a new stage.
AC-08
A retreat (backward stage move) is shown as a curved amber arrow (#C9A227) below the main pipeline track, with an arrowhead pointing left (←) toward the earlier stage. It includes a small label showing from_stage → to_stage.
AC-09
Retreat arrows also persist permanently alongside any later forward arrows covering the same span — both are visible simultaneously.
AC-10
On page load, all historical arrows are rendered correctly from the lead_stage_history table — the diagram accurately reflects every transition the lead has ever made.
AC-11
The current stage node is rendered as a larger filled circle with the stage icon and a pulsing ring animation.
AC-12
Hovering over a future stage node briefly tints the future arrows pointing to it in blue, previewing what the active arrow would look like if that transition were taken.
AC-13
Clicking any stage node opens a confirmation modal. The modal shows the current stage, the target stage, the stage description, a skip warning (if applicable), and a retreat warning (if moving backward).
AC-14
On modal Confirm, PATCH /leads/<id>/progress is called. On success, the diagram is replaced with the fully re-rendered SVG (including all history arrows) without a full page reload.
AC-15
The newly drawn active arrow plays the draw-on animation (strokes in from left to right over 0.45s) after the diagram is replaced in the DOM.
AC-16
Moving to On Hold shows the amber badge; arrows on the diagram remain in their pre-hold state — no new arrow is drawn for On Hold itself.
AC-17
Closed Won and Closed Lost terminal arrows are rendered in dark green / dark red respectively, with a 5px shaft and large arrowhead. No outbound arrow is drawn beyond the terminal node.
AC-18
Every stage transition (including retreats) is written to lead_stage_history with from_stage, to_stage, changed_by, changed_at, reason, and days_in_stage.
AC-19
The diagram is keyboard-navigable: Tab selects each node circle, Enter opens the confirmation modal.
AC-20
On screens narrower than 900px the SVG is horizontally scrollable inside the pipeline-wrapper container without breaking the page layout.

15. Open Questions
Should the stage history log be visible to the assigned rep on the Lead Detail page, or only to Admin? Current spec shows it only as a hover tooltip on the diagram.
Should the stale stage warning (14 days amber / 30 days red) also send a notification or create a task reminder for the assigned rep?
Should Closed Lost leads be re-openable by the assigned rep (not just Admin), or should terminal overrides always require Admin?
Should the On Hold duration appear in the stage history — i.e., should 'On Hold' appear as a row in lead_stage_history (and therefore generate an arrow), or is it excluded since it is not a true pipeline stage?
Should multi-location companies support a per-location PROGRESS stage (e.g., if a prospect has 3 sites, each site has its own pipeline position and its own arrow history)?
Should the Lead Dashboard list view show a mini version of the pipeline diagram (e.g., a condensed progress bar with a small arrow indicator) in each row, or only the text PROGRESS label?
For leads with many stage retreats (e.g., 5+ backward moves), the retreat arrows below the track could become visually crowded. Should there be a maximum number of visible retreat arrows, with older ones collapsed into a 'View full history' link?
Should the completed arrow color (#5A9E7A muted green) vary by stage group — e.g., blue arrows through Outreach stages, green through Active, purple through Closing — or remain a uniform muted green for all completed transitions?
